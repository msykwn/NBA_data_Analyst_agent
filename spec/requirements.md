# NBA データアナリストエージェント 要件定義書(仕様編)

## 1. 目的・背景

NBAのデータを分析できるチャット型AIエージェントを作成する。
NBAの公式サイトのスタッツページを見れば選手やチームの成績はわかるが、選手同士の比較や直近の成績の傾向などを
アナリストのように分析して回答してくれるサービスを作る。

本書は仕様面(機能・インターフェース・データ)についての要件を定義する。詳細設計や非機能要件の具体的な仕様は、実装段階で別途検討する。

### 1.1　裏目的
AWS Bedrock AgentCore の学習を主目的とし、AgentCore上で複数のエージェントを構築・運用する経験を積むことを狙いとする。


## 2. スコープ

- 対象: バックエンド + AWS Bedrock AgentCore 中心の構成
- 動作確認の単位: REST API エンドポイント経由でのリクエスト/レスポンス
- 対象外(将来フェーズ):
  - React フロントエンド(チャットUI)
  - AgentCore Gateway を経由した外部APIのツール化
  - 長期記憶(ユーザーの嗜好・要約の保持)
  - Identity / Code Interpreter / Browser / Observability などの他のAgentCore構成要素

## 3. システム構成概要

```
[クライアント (REST API リクエスト)]
        ↓
[Python バックエンド]
        ↓
[AWS Bedrock AgentCore Runtime]
        ├─ オーケストレーターエージェント
        │     ├─ データ取得エージェント ── (直接HTTP呼び出し) ── [NBAデータソース外部API]
        │     └─ 分析エージェント
        └─ Memory(短期記憶)
```

## 4. エージェント構成

### 4.1 全体構成方針

オーケストレーター(司令塔)エージェントが、ユーザーからの質問内容に応じて配下の専門エージェントへ処理を振り分け、その結果を統合してユーザーに回答する構成とする。

### 4.2 エージェントの役割分担

| エージェント | 役割 |
|---|---|
| オーケストレーターエージェント | ユーザーからの質問を受け取り、内容を解釈して適切な専門エージェントに処理を委譲する。専門エージェントの出力を統合し、最終回答を生成してユーザーに返す。 |
| データ取得エージェント | NBAデータソース(外部API)を呼び出し、選手・チーム・試合などの生データを取得する。Strands Agents で実装し、ステートレスに動作する。オーケストレーターから Agent as Tool として呼び出される。AgentCore Memory には接続しない。 |
| 分析エージェント | データ取得エージェントが取得したデータをもとに、比較・ランキング・要約などの分析を行い、回答に資する説明を生成する。Strands Agents で実装し、ステートレスに動作する。オーケストレーターから Agent as Tool として呼び出される。ツール関数は持たず、LLM のプロンプト処理のみで分析を行う。AgentCore Memory には接続しない。 |

### 4.3 エージェント間の連携イメージ

1. ユーザーからの質問をオーケストレーターエージェントが受け取る
2. オーケストレーターエージェントが質問内容を解釈し、必要に応じてデータ取得エージェントにデータ取得を依頼する
3. データ取得エージェントが外部APIからデータを取得し、結果を返す
4. 必要に応じて、取得したデータを分析エージェントに渡し、比較・ランキングなどの分析結果を生成する
5. オーケストレーターエージェントが各エージェントの結果を統合し、最終的な回答を生成してユーザーに返す

> 補足: 質問の種類によっては、データ取得エージェントの結果のみで回答が完結する場合(例: 「成績を教えて」のような単純な照会)と、
> 分析エージェントによる加工が必要な場合(例: ユーザー指定の比較・ランキング、「調子良い?」のような定型的な傾向質問)がある。
> ユースケースの分類は5.2節を参照。振り分けロジックの詳細は実装段階で検討する。

## 5. 想定ユースケース

### 5.1 想定ユーザー像

- **カジュアルなNBAファン**: 「推し選手の今シーズンの調子は?」のように、観戦をより楽しむための気軽な質問が中心。専門用語は最小限。
- **コアな分析好き**(開発者自身がイメージするユーザー): 選手間の比較やランキングの背景など、踏み込んだ問いを投げる。

いずれの場合も、エージェントは「公式スタッツページを見ればわかる数値」をただ転記するのではなく、アナリストのように整理・解説して返すことを目指す。

### 5.2 ユースケース一覧

エージェントが対応すべき主な質問・ユースケースは以下を想定する。

| ユースケース | 例 | 想定する回答の方向性 | 主に関与するエージェント |
|---|---|---|---|
| 成績・記録の検索 | 「○○選手の成績を教えて」「○○選手は今どこのチームに所属してる?」 | 該当選手の今シーズンの主要スタッツ(または指定された情報)を要約して回答する | データ取得エージェント(+ 簡単な要約は分析エージェント) |
| 指定対象の比較・ランキング分析 | 「○○選手と××選手の成績を比較して」「リーグ得点ランキング上位5人は?」 | ユーザーが指定した対象同士の主要スタッツを比較、またはランキング形式で回答する | データ取得エージェント + 分析エージェント |
| 定型的な傾向質問(定番の比較観点を自動選択) | 「○○選手は最近調子良い?」「○○選手は3Pが得意?」 | 質問の意図に応じて、エージェントが定番の比較観点を自動選択し、比較を交えて回答する(例: 「調子は?」→ 直近5試合 vs 今シーズン平均、「3Pは得意?」→ 本人の3P成績 vs 同ポジション平均) | データ取得エージェント + 分析エージェント |

> 上記はいずれも選手を主語にした例だが、**「選手」を「チーム」に置き換えた場合も同様の考え方で対応する**。例:
> - 成績・記録の検索: 「○○チームの直近の試合結果は?」「○○チームの地区順位は?」
> - 指定対象の比較・ランキング分析: 「○○チームと××チームの対戦成績を教えて」「直近5試合のチーム成績を先週と比べて」
> - 定型的な傾向質問: 「○○チームは最近調子良い?」→ 直近5試合 vs 今シーズン平均の成績を比較
>
> 上記以外の一般的なNBAに関する雑談・知識ベースの質問への対応は、本フェーズのスコープ外とする(オーケストレーターエージェントが自身の知識で簡易的に応答する、または対応しない、のいずれにするかは実装段階で検討)。

### 5.3 取り扱うデータ種別

データ取得エージェントが対象とするデータ種別は以下とする。

| データ種別 | 内容 | 備考 |
|---|---|---|
| 選手の個人スタッツ | 得点・リバウンド・アシスト・FG%・3P%等の平均/累計成績 | 最も基本的なユースケース |
| チーム成績・順位・対戦成績 | 勝敗数、地区/カンファレンス順位、特定チーム同士の対戦成績(ヘッドトゥヘッド) | - |
| 試合結果・スケジュール | 個別試合のスコア、直近の試合結果、今後の試合予定 | - |
| 所属チーム情報 | 選手の現所属チーム | プロフィール系情報は**必要最小限**に留め、現所属チームのみを対象とする(移籍履歴・経歴等の詳細プロフィールはスコープ外) |
| ポジション情報・ポジション別平均スタッツ | 選手のポジション、同ポジション選手群の平均スタッツ | 5.4節「定型的な傾向質問」(例:「3Pが得意?」→ 同ポジション平均との比較)で必要となる集計データ。外部API側でポジション別集計を提供しない場合、取得した個別選手データから算出する方式も実装段階で検討する |

### 5.4 分析の深さに関する方針

分析エージェントが生成する内容は、**「数値 + 簡単な傾向コメント」**を基本とする。比較を行うかどうか、また何と比較するかは、質問の種類によって以下のように切り分ける。

- **ユーザーが比較対象を明示した場合**: 指示された対象同士(例:「AとBを比較して」「上位5人を教えて」)で比較・ランキング分析を行う。
- **定型的な傾向質問の場合**: 「調子良い?」「3Pは得意?」のような定番フレーズに対しては、エージェントが質問の意図に応じた**定番の比較観点**を自動選択し、比較を交えて回答する(例: 「調子良い?」→ 直近5試合 vs 今シーズン平均、「3Pは得意?」→ 本人の3P成績 vs 同ポジション平均)。これらは「質問の意図を解釈した結果としてエージェントが選ぶ定番パターン」であり、リーグ全体や任意の対象との比較を無制限に行うものではない。
- **上記いずれにも該当しない単純な照会の場合**(例:「成績を教えて」): 比較は行わず、取得したデータに基づく簡単な要約(数値の整理 + 一言程度の傾向コメント)を返す。
- 怪我・出場時間・対戦カードの相性といった文脈を踏み込んで考察するアナリスト的な深い分析は、本フェーズでは対象外とする(将来の拡張候補)。

> 補足: 「定型的な傾向質問」をどこまでの範囲でパターンとして認識するか(定型句の一覧化、自然文の意図解釈など)は実装段階で検討する。

### 5.5 対象期間の扱い

- 対象期間は**ユーザーの指示に従う**ことを基本方針とし、指示がない場合は**今シーズン**をデフォルトとする。
- ユーザーが「先週」「直近5試合」のような**相対的な期間表現**を用いた場合も解釈して対応する。
  - この相対表現の解釈(=具体的な日付範囲や試合数への変換)は**オーケストレーターエージェントが質問解釈の段階で行う**。オーケストレーターが解釈結果(具体的な期間・件数のパラメータ)をデータ取得エージェントに渡すことで、データ取得エージェント側は解釈ロジックを持たず、与えられた条件でAPIを呼び出すことに専念できる構成とする。
- 5.4節の「定型的な傾向質問」(例:「調子良い?」→ 直近5試合 vs 今シーズン平均)には、エージェント側があらかじめ定めた**デフォルトの比較期間**(直近5試合など)が存在する。このデフォルト期間と、ユーザーが明示的に指定した期間(例:「最近1ヶ月で見て調子良い?」)が両立する場合は、**ユーザー指定の期間を優先**し、デフォルト期間は指定がない場合のフォールバックとして扱う。

## 6. データソース(NBAデータ取得API)

- データ取得元には **stats.nba.com(NBA公式統計サイト)** を採用する。アクセスには `nba_api`(stats.nba.comをラップしたOSSのPythonライブラリ)を経由する。
- 本フェーズでは、データ取得エージェントが `nba_api` を **直接呼び出す** 構成を前提とする(AgentCore Gateway を経由したMCP互換ツール化は将来の後工程として位置づける)。
- `nba_api` ツール関数のインターフェース方針:
  - 選手名またはチーム名(文字列)を受け取り、ツール内でIDに変換してからAPIを呼び出す
  - シーズンをパラメータとして受け取る(省略時は今シーズンをデフォルト)
  - 名前が見つからない場合はエラーではなく類似名の候補リストを返す
  - API呼び出し失敗時(タイムアウト・レート制限等)は最大3回リトライし、4xx系クライアントエラーはリトライせず即座にエラーを返す

## 7. インターフェース仕様

### 7.1 入出力形式

- クライアントからは **REST API エンドポイント** 経由でエージェントに質問を送信し、回答を受け取る。
- フロントエンド(React)は本フェーズの対象外。CLI/API レベルでの動作確認を優先する。

### 7.2 会話・セッション管理

- クライアントが `session_id`(または同等の会話ID)を発行・送信し、リクエストに付与することで会話を継続する**オーソドックスな構成**とする。
- バックエンドは `session_id` をもとに AgentCore Memory(短期記憶)と連携し、直近の会話コンテキストを保持・参照する。
- リクエスト/レスポンスの具体的なJSONスキーマ(エンドポイントパス、パラメータ名、ステータスコードなど)は実装段階で定義する。

## 8. Memory(記憶)に関する要件

- 本フェーズでは **短期記憶のみ** を要件に含める。
  - `session_id` に紐づく直近の会話コンテキストを保持し、同一セッション内での文脈維持(指示語の解決、直前の質問内容の参照など)に利用する。
- 長期記憶(ユーザーの嗜好や過去の要約の保持など)は、本フェーズのスコープ外とし、将来の拡張項目として位置づける。

## 9. ローカル開発環境

ローカル開発環境のセットアップ方針(`uv`・Python バージョン・AWS 認証方式など)は `spec/architecture.md` 9.1節を参照。
主要な依存パッケージは `pyproject.toml` で管理し、`uv sync` 一発で環境を再現できる構成とする。

AWS 認証には **aws-vault** を使用する。`~/.aws/credentials` に直接認証情報は書かれていないため、boto3 での直接プロファイル指定は機能しない。Python スクリプトの実行は以下の形式を用いる。

```bash
aws-vault exec piyo --no-session -- uv run python -m <module>
```

## 10. 非機能要件(方針レベル)

学習目的のプロジェクトであるため、本フェーズでは認証・エラーハンドリング・ログなどの非機能要件について **詳細仕様は定義せず、最小限の方針のみ** を記載する。具体的な実装方針は、実装段階で個別に検討する。

- **認証**: 外部API・AgentCoreへの接続に必要な認証情報(APIキー等)の管理方法は実装時に検討する。クライアント-バックエンド間の認証は本フェーズでは特に定めない。
- **エラーハンドリング**: 外部API呼び出し失敗時やエージェント処理エラー時の扱い(エラーレスポンスの形式、リトライの要否など)は実装時に検討する。
- **ログ**: 動作確認・デバッグに最低限必要なログ出力を行う方針とし、詳細な設計(ログレベル、出力先、フォーマット等)は実装時に検討する。
