# NBA データアナリストエージェント アーキテクチャ設計書

本書は `spec/requirements.md`(仕様編)を踏まえた、システム全体のアーキテクチャ・技術選定について定義する。
機能ごとの詳細な設計判断は `spec/features/` 配下の個別要件ファイル(開発完了後に `spec/adr/` へADR化)に記載する。本書は機能横断的なベースライン方針を扱う。

## 1. 全体構成

```
[クライアント (REST API リクエスト)]
        ↓
[AWS Bedrock AgentCore Runtime]  … エージェント実行 + API公開を兼ねる
        ├─ オーケストレーターエージェント ── (連携) ── [AgentCore Memory(短期記憶)]
        │     ├─ データ取得エージェント ── (Tool/Function経由でHTTP呼び出し) ── [NBAデータソース外部API]
        │     └─ 分析エージェント
        ↓
[Amazon Bedrock (Claude)]  … 各エージェントが利用する基盤モデル
```

要件定義書(`spec/requirements.md`)では「クライアント → Pythonバックエンド → AgentCore Runtime」という3層構成を想定していたが、本設計では **AgentCore Runtimeがエージェント実行とAPI公開の両方を兼ねる** 構成とする(詳細は2.2節)。

## 2. デプロイ単位・バックエンド構成

### 2.1 デプロイ単位: 単一Runtime内マルチエージェント

オーケストレーター・データ取得・分析の3エージェントは、**1つのAgentCore Runtimeの中にマルチエージェントとして同居**させる構成とする。

- **検討した選択肢**:
  - (A) 単一Runtime内マルチエージェント: オーケストレーターが配下のエージェントをサブエージェント(関数・クラス)として直接呼び出す
  - (B) エージェントごとに個別Runtime: 各エージェントを独立したAgentCore Runtimeとしてデプロイし、Runtime間をHTTP等で連携する
- **採用方針と理由**: (A) を採用。デプロイ・運用単位がシンプルで、AgentCore Runtimeの基本的な使い方を学ぶ最初のステップとして扱いやすい。Runtime間連携(B)は、AgentCore学習が一定進んだ将来フェーズでの拡張候補として位置づける。

### 2.2 Pythonバックエンドの位置づけ: AgentCore Runtimeに統合

要件定義書では「クライアント → Pythonバックエンド → AgentCore Runtime」という構成を示していたが、本設計では **AgentCore Runtimeがエージェント実行とAPI公開を兼ねる** ことで、両者を実質的に一体として扱う。

- **検討した選択肢**:
  - (A) AgentCore RuntimeがHTTPエンドポイントを提供し、クライアントから直接(または薄いプロキシ経由で)呼び出す
  - (B) FastAPI等で独立したAPIサーバを構築し、そこからAgentCore RuntimeのエージェントをSDK/API経由で呼び出す。`session_id` 管理やリクエスト/レスポンス整形をバックエンド側に持たせる
- **採用方針と理由**: (A) を採用。AgentCore Runtime自体がエージェント呼び出し用のエンドポイントを提供する仕組みを持つため、独立したAPIサーバを別途設けると責務が重複し、学習対象であるAgentCore Runtimeの動作確認という主目的から外れる。`session_id` の発行・付与や入出力形式は、AgentCore Runtimeが提供する規約に合わせる方針とし、具体的なJSONスキーマは実装段階で定義する。
- **影響**: 「Pythonバックエンド」という独立層は本設計上は登場せず、要件定義書 第3章の構成図とは異なる形になる。要件定義書側の構成図更新は、別途ユーザーの指示があった際に `merge-requirement` フローの中で整理する。

## 3. エージェントフレームワーク

### 3.1 採用: Strands Agents

マルチエージェントの実装には **Strands Agents**(AWSが公開するOSSのPythonエージェントフレームワーク)を採用する。

- **検討した選択肢**:
  - (A) Strands Agents: AWS製。AgentCoreとの統合が公式ドキュメント・サンプルで手厚くサポートされている
  - (B) LangGraph: マルチエージェント・ステートマシン型ワークフローで広く使われる。情報量は最大だが、AgentCoreとの接続部分は自前で設計する必要がある
  - (C) CrewAI: 「役割を持つエージェントのチーム」という設計思想が、要件定義書のオーケストレーター/データ取得/分析という役割分担と相性が良い。ただしAgentCoreとの統合事例はStrandsより薄い
  - (D) フレームワークを使わず自作: Bedrock Converse APIを直接呼び出し、エージェント間制御を自前実装する
- **採用方針と理由**: (A) を採用。本プロジェクトの裏目的は「AgentCore上で複数エージェントを構築・運用する経験を積むこと」であり、AgentCore側の公式ドキュメント・サンプルがそのまま使えるStrands Agentsは学習効率の観点で最も合理的。マルチエージェントパターン(Agent as Tool 等)も要件定義書の構成にそのまま当てはめやすい。
- **ロックインに関する検討**: Strands AgentsはOSSのPythonライブラリであり、AgentCore以外の環境(EC2/Lambda/ローカル等)でも動作し、Bedrock以外のモデルプロバイダにも対応するため、フレームワーク自体がAgentCoreにロックインするわけではない。AgentCore固有のロックインは「AgentCore Runtime/Memory のAPI・SDKを直接呼び出す」部分で生じるが、これはAgentCore学習という目的上、許容する。
- **マルチエージェントの実装パターン**: オーケストレーターエージェントが、データ取得エージェント・分析エージェントを「ツール(Agent as Tool)」として呼び出す構成を基本とする。具体的な実装パターンの詳細は実装段階で検討する。

## 4. 基盤モデル(LLM)

各エージェントが利用する基盤モデルは、Amazon Bedrock経由で **Claude(Anthropic)** を使用する。

- **理由**: ツール利用・オーケストレーションに強く、AgentCore・Strands Agentsのサンプル/ドキュメントでも採用例が多い。
- モデルサイズ(Haiku/Sonnet等)の使い分けは実装段階で検討する。コストを抑えたい処理(簡単な照会の要約等)と品質を重視したい処理(比較・分析等)で、エージェントごとに異なるモデルを使い分けることも視野に入れる。

## 5. 外部API(NBAデータソース)連携方式

データ取得エージェントが外部NBAデータソース(stats.nba.com、`nba_api` 経由)を呼び出す実装は、**Strands Agentsのツール(Tool/Function)として実装**する。

- **検討した選択肢**:
  - (A) フレームワークのツール機能としてPython関数を定義し、関数内で `nba_api` を使って stats.nba.com からデータを取得する。エージェント(LLM)が必要に応じてツールを呼び出す
  - (B) AgentCore Gateway経由でMCP互換のツール化を最初から導入する
- **採用方針と理由**: (A) を採用。要件定義書 第6章でも「データ取得エージェントが外部APIを直接HTTP呼び出しする」構成を前フェーズの前提としており、これと整合する。実装がシンプルで、ツール関数の単位を保ったまま将来的にAgentCore Gateway経由のMCP化へ移行する道筋も残せる。Gateway経由のMCP化(B)は、要件定義書 第10章にある通り将来フェーズの検討事項として位置づける。

## 6. AgentCore Memory(短期記憶)の連携

会話履歴の保持・参照は **オーケストレーターエージェントのみがAgentCore Memoryと連携** する構成とする。

- **検討した選択肢**:
  - (A) オーケストレーターのみがMemoryと連携し、配下のデータ取得・分析エージェントはステートレスに保つ
  - (B) 各エージェントがそれぞれMemoryにアクセスし、必要に応じて会話文脈を参照する
- **採用方針と理由**: (A) を採用。要件定義書 第5.5節にある通り、相対的な期間表現の解釈(「先週」「直近5試合」など)はオーケストレーターが質問解釈の段階で行い、解釈済みの具体的なパラメータを配下エージェントに渡す方針となっている。これと同様に、会話文脈の参照・利用もオーケストレーターに集約することで、責任分担が明確になり設計がシンプルになる。データ取得・分析エージェントは、与えられた条件に対して処理を行うことに専念できる。

## 7. 認証情報の管理

データ取得元として採用した stats.nba.com / `nba_api`(5節)はAPIキー等の認証を必要としないため、本フェーズでは外部NBA API向けの認証情報管理は**不要**となった。

- Secrets Managerなどによる認証情報管理の仕組みは、将来的に認証が必要なデータソース・外部サービスを追加する際の検討事項として位置づける。

## 8. インフラ構築・デプロイフロー

インフラ構築には **Terraform** を用いる。`.gitignore` にTerraformテンプレートが含まれている点とも整合する。

Terraformと、AgentCore Runtimeへのエージェントコードのデプロイは役割を分け、以下の使い分けとする。

- **Terraformで管理する範囲**: IAMロール、AgentCore Memory、その他関連AWSリソースなどの**インフラ**
- **別手段で行う範囲**: AgentCore Runtimeへの**エージェントコード自体のデプロイ**は、AgentCore CLI/SDK(例: `agentcore launch` 等)を使用する

- **検討した選択肢**:
  - (A) Terraform=インフラ、AgentCore CLI/SDK=エージェントコードデプロイ、と役割を分ける
  - (B) エージェントコードのデプロイも含めてすべてTerraformで一元管理する
- **採用方針と理由**: (A) を採用。AgentCore側のTerraformサポート状況は本設計時点では流動的であり、無理に一元化すると回り道が増える可能性がある。役割を分けることで、それぞれの公式ツールの使い方を素直に学べる。具体的なコマンド・手順は実装段階でAWS公式ドキュメントを確認の上、決定する。

## 9. ローカル開発・動作確認の進め方

開発は以下の順序で段階的に進める。

1. **ローカルでStrands Agents単体を動作確認**: AgentCoreにデプロイする前に、ローカル環境でStrands Agentsのエージェントロジック単体を動かす(Bedrock呼び出しは行う)。マルチエージェントの制御ロジックや外部APIツールの動作を、デバッグしやすい環境で先に固める。この段階では **AgentCore Memoryには接続しない**(6節の通り、Memory連携はオーケストレーターのみが担うため、まずはオーケストレーター・データ取得・分析の各エージェント間の連携ロジック自体の検証に集中する)。
2. **AgentCore Runtimeへのデプロイ・動作確認**: ローカルでの動作に問題がなければ、AgentCore Runtimeにデプロイし、REST API経由での動作確認を行う。
3. **Memory(短期記憶)の組み込み**: 上記の動作確認ができてから、AgentCore Memoryとの連携を組み込む(要件定義書「開発の進め方」と対応)。

- **理由**: AgentCore特有の環境構築・デプロイサイクルを待たずに、エージェントロジック自体の検証から始められるため、デバッグサイクルが速い。要件定義書「開発の進め方」(まず動作確認、その後Memory組み込み)とも整合する。

### 9.1 開発環境セットアップ方針

ローカルでのPython開発環境は以下の方針で構築する。

| 項目 | 採用方針 |
|---|---|
| パッケージ管理ツール | **uv**(`pyproject.toml` + `uv.lock` で依存関係を管理) |
| Pythonバージョン | **3.12** |
| ローカル実行時のAWS認証 | **AWS CLIプロファイル**(`~/.aws/credentials`)を利用し、boto3/Strands Agentsのデフォルト認証情報チェーンに解決させる。コード中に認証情報を直接記述しない |

- **パッケージ管理ツールの選定理由**: uvは高速なインストール・実行が可能で、AWS・Strands Agents関連の最新ドキュメント/サンプルでも採用例が増えている。学習用途として`pyproject.toml`ベースのモダンな構成に触れておく価値がある。
- **Pythonバージョンの選定理由**: 3.12はAWS Lambda/AgentCoreのランタイムでもサポートされる安定版であり、Strands Agentsなど主要ライブラリの対応も十分。
- **ローカル認証の選定理由**: AWS CLIプロファイルを使う方式は、認証情報をコード・環境変数・`.env`ファイルなどに記述する必要がなく、開発者のマシン上の標準的なAWS認証の仕組みにそのまま乗れるため、認証情報の漏洩リスクが小さい。AgentCore Runtime実行ロール(本番相当)とローカル開発用IAMユーザー/ロールは別物として扱う想定で、具体的な権限設計は実装段階で詰める。

## 10. ログ・観測性(Observability)

要件定義書 第9章では「最低限のログ出力」という方針レベルの記載のみとしていたが、本設計では将来的な拡張を見据え、以下を視野に入れる。

- **基本方針**: AgentCore Runtimeが標準で出力するCloudWatch Logsの仕組みをそのまま活用し、アプリケーションコード側では最低限の `print`/`logging` 出力を行う
- **将来の拡張候補**: AgentCore Observability(トレース・メトリクス等)の導入。マルチエージェント構成では、どのエージェントがどう処理を委譲し合ったかを追跡できると、デバッグ・学習の両面で有用なため、状況を見て検討する

具体的なログレベル・出力先・トレース設計などは実装段階で検討する。
