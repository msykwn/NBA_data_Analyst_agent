## 概要

ローカル環境でエージェントを動かす際に、オーケストレーターからサブエージェント・ツールへの呼び出しの流れをターミナルにリアルタイムで表示する。どのエージェントが呼ばれたか・どのツールが呼ばれたか・それぞれの入出力を色付き・インデントで階層表示し、エージェントの内部動作を可視化する。

**スコープ（やること）:**
- オーケストレーター → サブエージェント → ツールの呼び出し階層をターミナルに表示
- 各ステップの入力・出力を表示
- 色付き・インデントで階層がわかりやすい表示フォーマット

**スコープ（やらないこと）:**
- ファイルへのログ保存
- ブラウザUIでの可視化（Jaeger、Langfuse等）
- AWS CloudWatch等クラウドへの送信
- エラー時のアラート

## 背景・課題

ローカル CLI でエージェントと会話しながら AgentCore の学習を進める方針（2026-06-11）に変更したことに伴い、エージェントの内部動作を観察したいニーズが生まれた。

現状は CLI からプロンプトを送って最終回答が返ってくるだけで、オーケストレーターが本当にサブエージェントに処理を渡しているか・どのツールが呼ばれているかが見えない。デバッグ目的というより、複数エージェント構成の動作を学習として理解するために可視化したい。

**前提・制約:**
- ローカル環境のみ対象（AgentCore Runtime 上での可視化は対象外）
- エージェントフレームワークは Strands Agents を使用
- Strands Agents がトレース・コールバックの仕組みを提供しているかどうかが実装方式の鍵になる

## 要件

### 機能要件

- ユーザーのプロンプトへの回答が完了した後に、その会話のトレースをターミナルに表示する
- 表示する情報:
  - どのエージェントが呼ばれたか（オーケストレーター → サブエージェントの流れ）
  - 各エージェントがどのツールを呼んだか
  - それぞれの入力・出力（全量表示）
- 呼び出しの階層構造がわかるよう、インデントと色付きで表示する
- ローカル環境でのみ動作すればよい

### 非機能要件

- 特になし（学習目的のため厳密な要件は設けない）

## 検討した選択肢

### 実装方式

#### 選択肢A: Strands Agents の `add_hook` を使う（採用）
- メリット: フレームワーク公式の仕組み。追加ライブラリ不要。`BeforeToolCallEvent` / `AfterToolCallEvent` / `BeforeMultiAgentInvocationEvent` / `AfterMultiAgentInvocationEvent` 等、必要なイベントが揃っている
- デメリット: フックで収集したデータを自前でフォーマットして表示する必要がある

#### 選択肢B: OpenTelemetry（OTEL）でトレースを収集する
- メリット: 標準的なオブザーバビリティの学習になる。Strands Agents は OTEL 対応済み
- デメリット: Jaeger 等のバックエンドをローカルで別途立ち上げる必要がある。ターミナル出力には向かない

## 採用した方針と理由

- Strands Agents の `add_hook`（選択肢A）を採用。ターミナルへの出力・追加ライブラリ不要という要件に最も合致する。OTEL は今フェーズには重く、ターミナル出力に向かないため見送り

## 状態

実装・動作確認完了（2026-06-11）

## この決定によって生じる影響

**良い影響:**
- オーケストレーター → サブエージェント → ツールの呼び出し階層がターミナルで一目でわかるようになり、複数エージェント構成の動作理解が深まる
- Strands Agents のフックシステムを学べる
- 追加ライブラリ・インフラ不要でシンプルに導入できる

**リスク・懸念点:**
- フックで取れる情報量や粒度が Strands Agents の実装に依存するため、実装してみないとわからない部分がある（実装中に判明した場合はこのセクションを更新する）
- 入出力を全量表示するため、長いレスポンスの場合にターミナルが見づらくなる可能性がある。必要であれば省略オプションを後から追加する
