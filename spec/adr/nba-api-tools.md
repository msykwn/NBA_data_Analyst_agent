## 概要

Strands Agents の `@tool` デコレータを使い、データ取得エージェントが呼び出す nba_api ツール関数群を実装する。
対象データ種別は `spec/requirements.md` 5.3節に定義された全5種類。ツール関数はデータ取得エージェント(LLM)から呼び出され、stats.nba.com からデータを取得して返す。

## 背景・課題

- データ取得エージェントが NBAデータを取得するには、stats.nba.com へのアクセスを抽象化したツール関数が必要
- `spec/architecture.md` 5節の方針として「Strands Agents のツール(Tool/Function)として Python 関数を定義し、エージェント(LLM)が必要に応じて呼び出す」構成を採用している
- `nba_api` は stats.nba.com をラップした OSS の Python ライブラリで、APIキー不要で利用できる(`spec/requirements.md` 6節・`spec/architecture.md` 7節)
- ツール関数を Strands Agents のツールとして実装することで、将来的に AgentCore Gateway 経由の MCP 化へ移行する道筋も残せる(`spec/architecture.md` 5節)

## 要件

### 機能要件

**対象データ種別(全5種)**

- 選手の個人スタッツ取得(得点・リバウンド・アシスト・FG%・3P% 等の平均/累計)
- チーム成績・順位・対戦成績取得
- 試合結果・スケジュール取得
- 所属チーム情報取得(現所属チームのみ)
- ポジション情報・ポジション別平均スタッツ取得

**入力**

- 選手名またはチーム名(文字列)を受け取り、ツール内で ID に変換してから nba_api を呼び出す
- シーズン(例: `"2024-25"`)をパラメータとして受け取る。省略時は今シーズンをデフォルトとする
- 上記の入力インターフェースは全5種のツールで統一する

**出力・振る舞い**

- 正常系: 該当データを dict または構造化データとして返す
- 名前が見つからない場合: エラーではなく、類似名の候補リストを返す(全5種共通)
- API 呼び出し失敗時(タイムアウト・レート制限等): 最大3回リトライし、それでも失敗した場合はエラーを上位に伝播する(全5種共通)

### 非機能要件

- 特になし(学習目的のプロジェクトのため)

## 検討した選択肢

### 選択肢A: 名前→ID変換をツール内で行う
- メリット: LLM(エージェント)が選手IDを知らなくても自然な名前で呼び出せる。ツールの使いやすさが上がる
- デメリット: ツール内のロジックが増える。名前の表記揺れ(フルネーム/略称/日本語表記等)への対応が必要

### 選択肢B: 選手IDを直接受け取る
- メリット: ツールがシンプル。ID解決をLLM側に委ねるため、ツール自体の責務が明確
- デメリット: LLMが正確なIDを知っている必要があり、現実的には難しい。プロンプト設計が複雑になる

## 採用した方針と理由

選択肢Aを採用。LLM から自然な名前で呼び出せる方が、エージェントの実装がシンプルになり学習目的にも合っている。
名前の表記揺れへの対応コストはあるが、見つからない場合に候補リストを返す仕組みで補う方針とする。

## 状態

採用・完了(2026-06-10)

## この決定によって生じる影響

### 良い影響
- データ取得エージェント(LLM)がIDを意識せず自然な名前でツールを呼び出せるため、エージェントのプロンプト設計がシンプルになる
- ツール関数の入出力インターフェースを全5種で統一することで、データ取得エージェントからの呼び出し方が一貫する
- Strands Agents のツールとして実装するため、将来の MCP 化への移行パスが残る

### 悪い影響・リスク・前提条件
- 名前→ID変換のロジックが必要になり、表記揺れ(スペルミス・略称・日本語等)への対応が実装上の課題になりうる
- 候補リストを返す仕組みの具体的な実装(nba_api の検索機能の範囲・精度)は実装段階で確認が必要
- stats.nba.com はレート制限が非公開であり、リトライ間隔の設定は実装段階で調整が必要

## 仕様変更の経緯

### 2026-06-10: ポジション別平均スタッツの取得方法を変更
- 変更前: `LeagueDashPlayerStats` の結果を `PLAYER_POSITION` 列でフィルタリングする設計
- 変更理由: `LeagueDashPlayerStats` には `PLAYER_POSITION` 列が存在しないことが実装時に判明
- 変更後: `LeagueDashPlayerStats` の `player_position_abbreviation_nullable` パラメータ("G"/"F"/"C")でAPIレベルで絞り込む方式を採用。コードがシンプルになり、API呼び出し1回で完結する
