## 概要

ユーザーの質問を受け取り、`data_agent`・`analysis_agent` を Agent as Tool として呼び出し、各エージェントの結果を統合して最終回答を生成するオーケストレーターエージェントを実装する。

**スコープ**: ローカルでの動作確認まで。AgentCore Runtime へのデプロイ・Memory 連携は本フェーズ外。

## 背景・課題

`data_agent`(データ取得)と `analysis_agent`(分析)は実装済みだが、現状それぞれを単独で動かすことしかできない。ユーザーの質問に応じて「どちらを使うか」「どの順で呼ぶか」を制御する司令塔が存在しないため、マルチエージェントとして連携できていない。

オーケストレーターを実装することで、ユーザーが直接エージェントを意識せず自然言語で質問を投げるだけで、適切なエージェントへの振り分け・結果の統合・最終回答の生成が行われる構成が完成する。

また、「先週」「直近5試合」などの相対的な期間表現の解釈もオーケストレーターが担う(`spec/requirements.md` 5.5節)。

## 要件

### 機能要件

- 以下3パターンの質問に対して、適切にエージェントを呼び分け・統合して回答できること
  1. **単純な照会**（例：「レブロンの今シーズン成績は？」）→ `data_agent` のみ呼び出し
  2. **明示的な比較・ランキング**（例：「レブロンとカリーを比較して」）→ `data_agent` → `analysis_agent` の順で呼び出し
  3. **定型的な傾向質問**（例：「レブロンは最近調子良い？」）→ `data_agent` → `analysis_agent` の順で呼び出し
- 「先週」「直近5試合」などの相対的な期間表現を解釈し、具体的なパラメータとして配下エージェントに渡すこと
- マルチエージェント間の処理フロー（どのエージェントが何を処理したか）がログで追えること

### 非機能要件

- 特になし（学習目的のため詳細仕様は定めない）

## 検討した選択肢

### エージェント呼び分けの判断方法

#### 選択肢A: システムプロンプトで指示し、LLMが判断する
- メリット: 実装がシンプル。Strands Agents の Agent as Tool の仕組みにそのまま乗れる。質問パターンが増えても柔軟に対応できる
- デメリット: LLMの判断に依存するため、呼び分けの精度が保証できない。デバッグが難しい

#### 選択肢B: Pythonコードで明示的にルーティング条件を書く
- メリット: 呼び分けロジックが明確で予測可能
- デメリット: 質問パターンの分類をコードで管理する必要があり、柔軟性が低い。自然言語の多様な表現に対応しにくい

### 相対的な期間表現の解釈方法

#### 選択肢A: システムプロンプトで「解釈してパラメータ変換せよ」と指示し、LLMに任せる
- メリット: 実装がシンプル。多様な表現に柔軟に対応できる
- デメリット: 解釈の精度が保証できない。「先週」の定義がLLMによってぶれる可能性がある

#### 選択肢B: Pythonコードとして日付計算ロジックを実装し、ツール関数として持たせる
- メリット: 解釈が正確で一貫性がある
- デメリット: 実装コストが高い。想定外の表現に対応できない

#### 選択肢C: 今回はスコープ外として文字列をそのまま渡す
- メリット: 実装コストゼロ
- デメリット: 配下エージェントが「先週」を解釈できず、正しいデータを取得できない

### Agent as Tool の実装方法

#### 選択肢A: `agent_tool` デコレータで既存 Agent インスタンスをツール登録
- メリット: Strands Agents の公式パターン。既存の `data_agent`・`analysis_agent` をそのまま再利用できる。実装がシンプル
- デメリット: Strands Agents の内部動作に依存するため、バージョンアップ時の挙動変化に注意が必要

#### 選択肢B: ラッパー関数として `@tool` デコレータでツール化
- メリット: 呼び出しインターフェースを自前でコントロールできる
- デメリット: 実装量が増える。既存エージェントの再利用性が下がる

## 採用した方針と理由

- **エージェント呼び分けの判断**: 選択肢A(LLMが判断)を採用。Strands Agents の Agent as Tool パターンに自然に乗れ、実装がシンプル。呼び分け精度が足りない場合は後でルーティングロジックを追加する方針で許容する
- **相対的な期間表現の解釈**: 選択肢A(LLMに任せる)を採用。まずLLMの解釈力がどの程度通用するかを動作確認で検証し、精度が不十分な場合に日付計算ロジックのツール化(選択肢B)へ切り替える
- **Agent as Tool の実装**: 選択肢A(`agent_tool` デコレータ)を採用。Strands Agents の公式パターンであり、既存エージェントをそのまま再利用できるため学習コストが低い

## この決定によって生じる影響

### 良い影響
- マルチエージェント構成が完成し、ユーザーが自然言語で質問するだけで適切な回答が得られる
- LLMに判断を委ねる構成のため、実装がシンプルで質問パターンの追加にも対応しやすい
- `data_agent`・`analysis_agent` の再利用性が高い（Agent as Tool パターンにより既存実装をそのまま活用）

### 悪い影響・トレードオフ
- エージェント呼び分けと期間表現の解釈をともにLLMに委ねるため、精度が保証できない。動作確認で精度を見極め、必要であれば追加実装が必要になる
- AgentCore Memory との連携は本フェーズ外のため、セッションをまたいだ会話文脈の維持はできない

### 実装中に発覚した制約と対応

**`_AgentAsTool` の並列呼び出し競合**
- `Agent.as_tool()` は呼び出し時に `_AgentAsTool` インスタンスを生成するが、`tools=[data_agent.as_tool(), ...]` の形でモジュールロード時に一度評価すると固定される
- `_AgentAsTool.stream()` は `self._lock.acquire(blocking=False)` で排他制御しており、ロック取得失敗時は `status="error"` を即時返す（リトライなし）
- オーケストレーターが同一エージェントを並列呼び出ししようとすると `agent is already processing a request` 警告が発生し、LLM がリカバリしなければデータ欠落が起きる
- **対応**: (1) `create_orchestrator()` ファクトリ関数を導入しリクエストごとに `as_tool()` を再評価、(2) システムプロンプトで同一エージェントへの並列呼び出しを明示的に禁止、(3) `SequentialToolExecutor` でオーケストレーターレベルのツール実行を直列化、の3点を合わせて対処

**orchestrator の会話履歴汚染**
- `orchestrator` インスタンスを使い回すと `messages` が蓄積し、後の質問に前の会話コンテキストが混入する
- `preserve_context=False`（デフォルト）はサブエージェント側のリセットのみ制御し、orchestrator 自身には効かない
- **対応**: `create_orchestrator()` でリクエストごとに新規インスタンスを生成することで解消

**SYSTEM_PROMPT への現在日付の注入**
- 期間表現（「先週」「最近1ヶ月」）の解釈をLLMに委ねる際、LLM は実行日を知らないためトレーニングデータ依存になり日付がずれる可能性がある
- **対応**: `_build_system_prompt()` を関数化し、`date.today().isoformat()` を毎回注入する構造に変更
