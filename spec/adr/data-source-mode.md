## 概要

NBA データ取得のバックエンドを、環境に応じて `nba_api`（`stats.nba.com`）と事前に用意したデータセットの2モードで切り替えられるようにする。ツールのインターフェース（関数名・引数・戻り値の構造）は変えず、バックエンド実装だけを差し替える設計にする。

- **`nba_api` モード**: ローカル開発用。メインの利用形態
- **`dataset` モード**: AgentCore Runtime 用の簡易代替。ローカルで `nba_api` を使って事前取得・保存したデータセットファイルから返す

**スコープ（やること）:**
- `nba_api` モードと `dataset` モードの2実装を用意し、環境変数で切り替えられるようにする
- 既存の `@tool` 関数（`get_player_stats` 等6関数）のインターフェースを維持する
- データセット生成スクリプトを用意する（`nba_api` を使ってローカルで実行）

**スコープ（やらないこと）:**
- データセットの自動更新・定期更新
- フロントエンド側のモード切り替えUI
- データセットに含まれない選手・チームへの対応（対象外である旨を返すのみ）

## 背景・課題

AgentCore Runtime へのデプロイ後、`nba_api`（`stats.nba.com`）への HTTP リクエストが毎回 `ReadTimeout` でタイムアウトすることが判明した。ローカル環境（自宅 IP）では正常動作するため、`stats.nba.com` が AWS のパブリック IP からのアクセスをブロックしていると判断した（詳細は `spec/adr/agentcore-deployment.md` 参照）。

**なぜ dataset モードを選んだか:**
Web 検索（Tavily 等）による代替も検討したが、以下の理由で却下した（詳細は「検討した選択肢」セクション参照）。ローカルで `nba_api` を使って事前取得したデータセットをファイルとして持つ方式であれば、追加サービス・追加コストなしに AgentCore Runtime 上でも動作し、データ精度も担保できる。

**なぜ切り替え式にするか:**
- ローカル環境では `nba_api` が正常に動作し、NBA 公式データとして精度・構造が安定しているため、ローカルでは `nba_api` をそのまま使いたい
- AgentCore の学習という本来の目的を維持しつつ、Runtime 上でも最低限動作する状態を作りたい
- どちらか一方に固定するのではなく、実行環境に応じて切り替えられる設計にしておく

**前提・制約:**
- `nba_api` モードはローカル専用（AgentCore Runtime では動作しない）
- `dataset` モードは AgentCore Runtime・ローカル双方で動作する
- データセットの対象は「得点ランキング上位30名」と「セルティックス全選手」に限定
- データセットは固定（定期更新なし）。データが古くなっても許容する

## 要件

### 機能要件

- 環境変数 `DATA_SOURCE_MODE` でデータソースを切り替えられる
  - `nba_api`: `nba_api` ライブラリ（`stats.nba.com`）を使って取得する（既存実装）。デフォルト
  - `dataset`: 事前取得済みのデータセットファイルから返す（新規実装）
  - 未設定または不正な値の場合は `nba_api` をデフォルトとして使用する
- 既存の `@tool` 関数（`get_player_stats`・`get_player_recent_games`・`get_player_info`・`get_team_standings`・`get_team_game_log`・`get_position_avg_stats`）の関数名・引数・戻り値の構造はモードによらず変えない
- `dataset` モードでカバー対象外の選手・チームを問い合わせた場合は、対象外である旨を `found: False` で返す
- データセット生成スクリプトを `scripts/generate_dataset.py` として用意する
  - `nba_api` を使ってローカルで実行し、`data/` ディレクトリに JSON ファイルとして保存する
  - 対象: 得点ランキング上位30名・セルティックス全選手の各種スタッツ

### 非機能要件

- 特になし（学習目的のため、パフォーマンス・精度の厳密な要件は設けない）

## 検討した選択肢

### データソース代替案の検討

#### 選択肢A: Tavily 等の Web 検索 API 経由
- メリット: データが常に最新。カバー範囲が広い
- デメリット: API キーが必要。検索結果から構造化データを抽出する精度に依存し、`nba_api` のような正確な数値が返る保証がない。Tavily サーバーへの接続可否も事前確認が必要（前回の教訓）

#### 選択肢B: 事前取得データセット（採用）
- メリット: 追加サービス・コスト不要。データ精度は `nba_api` と同等（同じソースで生成するため）。AgentCore Runtime での動作に外部依存がない
- デメリット: カバー範囲が限定的（上位30名＋セルティックス）。データが古くなる。データセット生成スクリプトの整備が必要

### 実装方式の検討

#### 選択肢A: ファイルを分けて環境変数で import を切り替える（採用）

```
src/tools/
  nba_tools_nba_api.py    # 既存実装をリネーム
  nba_tools_dataset.py    # 新規実装
  nba_tools.py            # DATA_SOURCE_MODE を見てどちらかを re-export
```

- メリット: 各実装がファイルとして独立しておりシンプル。呼び出し側（`data_agent.py`）の変更不要
- デメリット: `nba_tools.py` でのモジュールレベル条件分岐 import がやや間接的。ただし Python では一般的なパターンであり許容範囲

#### 選択肢B: 同一ファイル内で実装を分けて切り替える
- メリット: ファイル構成を変えなくて済む
- デメリット: 1ファイルが肥大化する

#### 選択肢C: tools ディレクトリをパッケージ化する
- メリット: 将来3つ目のデータソースを追加しやすい
- デメリット: 今回の変更量が最も多い

## 採用した方針と理由

- **データソース代替**: 事前取得データセット（選択肢B）を採用。Web 検索は精度・動作保証の不確実性があり、学習目的の簡易代替としてはコストに見合わないと判断した
- **実装方式**: ファイル分割＋環境変数 import 切り替え（選択肢A）を採用。`nba_tools.py` をエントリーポイントとし、`DATA_SOURCE_MODE` の値に応じて `nba_tools_nba_api.py` または `nba_tools_dataset.py` を re-export する。呼び出し側（`data_agent.py`）は変更不要。選択肢Bは肥大化、選択肢Cはオーバーエンジニアリングのため却下

## この決定によって生じる影響

**良い影響:**
- AgentCore Runtime 上でもエージェントが動作するようになり、フロントエンド接続・Memory 組み込みへの道が開ける
- `nba_api` 実装を残すことでローカル開発時の精度・安定性を維持できる
- 環境変数1つで切り替えられるため、デプロイ設定の変更だけで対応できる

**リスク・懸念点:**
- **カバー範囲の限定**: `dataset` モードは上位30名＋セルティックスのみのため、対象外選手への質問には答えられない。ユーザーへの案内が必要
- **データの陳腐化**: データセットは固定のため、シーズンが進むにつれて実態と乖離する。学習目的として許容するが、将来的に再生成の運用が必要になる可能性がある
- **直近試合データの制約**: `get_player_recent_games` や `get_team_game_log` は「直近N試合」を返すが、データセット生成時点の試合ログが固定されるため、「最新の試合」としての意味が薄れる

### 2026-06-11: Web 検索モードからデータセットモードに方針変更

- **変更前**: `nba_api` モードと Strands Agents 組み込み Web 検索モードの2択で設計
- **変更理由**:
  - Strands Agents に「組み込み Web 検索」ツールは存在せず、`strands-agents-tools` の `tavily` ツールが最も近い選択肢だった
  - Tavily はデータを代行取得するが、NBA スタッツのような構造化データを精度よく返せる保証がない
  - `http_request` ツールも検討したが、`stats.nba.com` へのブロックと同じ問題が残る
  - AgentCore の学習が主目的のため、データ精度を犠牲にしてまでクラウド対応を優先する必要はないと判断
- **変更後**: `nba_api` モード（ローカル用・メイン）と `dataset` モード（AgentCore Runtime 用・簡易代替）の2択に変更
