## 概要

データ取得エージェントが取得した生データをもとに、比較・ランキング・要約などの分析を行い、
回答に資する説明を生成する分析エージェントを Strands Agents で実装する。
オーケストレーターから「Agent as Tool」として呼び出せる形で定義する。

## 背景・課題

- マルチエージェント構成において、データ取得エージェントが取得した生データをそのままユーザーに返すだけでは、
  「アナリストのような回答」というプロダクト要件を満たせない
- 比較・ランキング・傾向コメントなどの「分析」処理を担う専門エージェントが必要
- データ取得エージェント(#4)が完成し、生データの出力形式が確定したため、分析エージェントの実装に着手できる状態になった

## 要件

- `src/agents/analysis_agent.py` に Strands Agents で分析エージェントを実装する
- 以下の分析パターンに対応する(spec/requirements.md 5.4節):
  - **単純な照会**: 比較なしで、取得データの簡単な要約(数値整理 + 一言傾向コメント)を返す
  - **明示的な比較・ランキング**: ユーザーが指定した対象同士の主要スタッツを比較・ランキング形式で回答する
  - **定型的な傾向質問**: 「調子良い?」→ 直近5試合 vs 今シーズン平均、「3Pが得意?」→ 本人3P成績 vs 同ポジション平均、などの定番パターンで比較を交えて回答する
- オーケストレーターから Agent as Tool として呼び出せるインターフェースで定義する
- 分析の深さ: 「数値 + 簡単な傾向コメント」が基本。怪我・対戦カード相性などの深い考察は対象外
- AgentCore Memory には接続しない(ステートレス)
- `uv run python src/agents/analysis_agent.py` で単体動作確認できること

## 検討した選択肢

### 選択肢A: ツールなし・プロンプトのみで分析
- メリット: シンプル、実装コストが低い
- デメリット: 受け取るデータ構造が複雑になると、プロンプトだけでは意図通りの分析を安定して行えない可能性がある

### 選択肢B: Pythonツール関数で数値計算を補助し、LLMは解釈・コメント生成に集中
- メリット: 平均計算・比較演算などをコードで確実に行えるため、数値の正確性が担保しやすい
- デメリット: ツール関数の実装コストが増える

## 採用した方針と理由

選択肢Aを採用。

本フェーズでは分析の深さを「数値 + 簡単な傾向コメント」に限定しており、
データ取得エージェントがすでに整形済みのデータを返す設計のため、
LLMの自然言語処理能力で十分に対応可能と判断した。
ツール関数の追加は、数値計算の複雑さが増した段階で改めて検討する。

## 状態

採用（実装中）

## この決定によって生じる影響

- **良い影響**: 実装がシンプルで、プロンプト調整だけで分析の振る舞いを柔軟に変えられる
- **悪い影響・制約**: LLMが数値計算を誤る可能性があるため、ランキング順位や差分の正確性は完全には保証できない。本フェーズの分析深度では許容範囲と判断する
